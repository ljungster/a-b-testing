A/B Testing

A/B Testing is one of the cheapest ways to determine whether UI/UX changes impact your site's users. To practice this essential skill, I made two simple websites and tracked users while they completed a task: put $150 worth of cacti merchandise in their cart.

Interface A

I kept interface A unchanged from my UI/UX course's original template. This interface acted as my control for this A/B test.

Go to app
Interface B

For interface B, I made 3 significant changes: I opted for a row layout, increased the font size of the prices, and removed the item descriptions.

Go to app

Hypotheses for Time To Completion

Null hypothesis: There is no difference between Site A’s time to completion and Site B’s time to completion.

Alternative hypothesis: Site A’s time to completion is greater than Site B’s time to completion.

Hypotheses for Return Rate

Null hypothesis: There is no difference between Site A’s return rate and Site B’s return rate.

Alternative hypothesis: Site A’s return rate is greater than Site B’s return rate.

Data Collection Process

I was able to collect my results via other students in my course. We all tested each other's sites, with the same goal of adding > $150 of merchandise to our carts.

Results Infographic

Broad Takeaways

I failed to reject the null hypothesis for either Time to Completion or Return Rate. One of the biggest limitations of this test was the small sample size of users. That required any difference to be dramatic in order to be statistically significant. My results seem to imply that layout does not have as much of an impact as I think it does. I find that notion hard to believe.
